Git & Github

What it is and why it's important for you to learn how to use it.

What is Git?

Git is a version control system - think of it as a time machine for your code and files.

Why Version Control Matters in Cybersecurity:

  • Audit trail: See who changed what security configuration and when
  • Rollback: Undo changes if a security script breaks something
  • Collaboration: Multiple team members can work on incident response playbooks
  • Backup: Your security tools are safely stored and recoverable

Git vs GitHub:

  • Git = The tool that runs on your computer (tracks changes)
  • GitHub = A website where you host your Git repositories online

Key Concepts You Need to Know

Repository (Repo)

A project folder tracked by Git. Contains all your files and the complete history of changes.

Example: Your penetration testing scripts collection

Commit

A snapshot of your project at a specific point in time. Like saving a checkpoint.

Example: "Fixed vulnerability scanner bug"

Branch

A parallel version of your repository. Lets you work on new features without breaking the main code.

Example: Testing a new exploit module

Clone

Making a copy of a repository from GitHub to your local machine.

Example: Downloading OWASP ZAP source code

Setting Up Git

1. Check if Git is installed:

git --version

2. Install Git (if needed):

  • Mac/Linux: Usually pre-installed, or use brew install git (Mac)
  • Windows: Download from git-scm.com (choose Git Bash during install)

3. Configure Git with your identity:

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

This info appears in your commits so people know who made changes.

The Basic Git Workflow

The Four-Step Process:

  1. Make changes to your files (write code, edit configs)
  2. Stage changes with git add (tell Git what to track)
  3. Commit changes with git commit (save a snapshot)
  4. Push changes with git push (upload to GitHub)

Essential Commands:

git status # See what files have changed
git add . # Stage all changed files
git commit -m "message" # Commit with a message
git push # Upload to GitHub

Hands-On: Creating Your First Repository

Method 1: Start on GitHub (Recommended for beginners)

  1. Go to github.com and sign up/login
  2. Click the + icon ‚Üí "New repository"
  3. Name it (e.g., "security-scripts")
  4. Check "Add a README file"
  5. Click "Create repository"

Method 2: Start Locally

mkdir my-security-project
cd my-security-project
git init # Initialize Git in this folder
git add .
git commit -m "Initial commit"

Then create a repo on GitHub and connect them (we'll cover this later)

Cloning a Repository

Want to work with someone else's code or your own repo from GitHub? Use git clone!

Example: Clone a security tool

Let's say you want to download a popular security tool from GitHub:

git clone https://github.com/username/security-tool.git

This creates a folder called security-tool with all the code inside.

Always review code before running it! Cloned repositories can contain malicious code. Use:

ls -la # List all files including hidden

Making Changes and Commits

Scenario: You're updating a security scanning script

# 1. Check what's changed
git status

# 2. See exactly what changed in each file
git diff

# 3. Stage your changes
git add scanner.py # Add specific file
git add . # Add all changed files

# 4. Commit with a descriptive message
git commit -m "Added SSL certificate validation check"

Writing Good Commit Messages:

  • ‚úÖ "Fixed SQL injection vulnerability in login form"
  • ‚úÖ "Added rate limiting to API endpoints"
  • ‚ùå "Fixed stuff" (too vague!)
  • ‚ùå "Update" (doesn't explain what was updated)

Pushing Your Changes to GitHub

Once you've committed changes locally, push them to GitHub:

git push origin main

Breaking it down:

  • git push = upload changes
  • origin = the default name for your GitHub repository
  • main = the branch you're pushing to (default branch)

First Time Pushing?

If this is your first push, you might need to set the upstream:

git push -u origin main

The -u flag sets the upstream, so next time you can just type git push

Pulling Changes from GitHub

Your teammate updated the incident response playbook on GitHub. Get those changes:

git pull origin main

This downloads and merges changes from GitHub into your local repository.

‚ö†Ô∏è Best Practice:

Always pull before you push! This prevents conflicts:

git pull # Get latest changes
# Make your changes
git add .
git commit -m "Your message"
git push # Upload your changes

Branching: Work Safely in Parallel

Branches let you experiment without breaking the main code.

Cybersecurity Use Cases:

  • Testing a new exploit without breaking your main toolkit
  • Developing different attack scenarios for red team exercises
  • Multiple analysts working on different parts of a security audit

Basic Branch Commands:

git branch # List all branches
git branch feature-new-scanner # Create new branch
git checkout feature-new-scanner # Switch to branch
git checkout -b feature-new-scanner # Create AND switch

# Make changes, commit them
git add .
git commit -m "Built new vulnerability scanner"
git push origin feature-new-scanner # Push branch to GitHub

Merging Branches

Once your feature is ready, merge it back into the main branch:

# Switch to main branch
git checkout main

# Pull latest changes
git pull

# Merge your feature branch
git merge feature-new-scanner

# Push merged code
git push

Better Way: Pull Requests on GitHub

Instead of merging locally, use GitHub Pull Requests (PRs):

  1. Push your branch to GitHub
  2. Go to GitHub ‚Üí "Pull requests" ‚Üí "New pull request"
  3. Select your branch to merge into main
  4. Add description, request review from teammates
  5. Merge when approved!

Handling Merge Conflicts

Sometimes Git can't automatically merge changes. This happens when two people edit the same line.

What a Conflict Looks Like:

<<<<<<< HEAD
allowed_ips = ["192.168.1.0/24"]
=======
allowed_ips = ["10.0.0.0/8"]
>>>>>>> feature-branch

How to Resolve:

  1. Open the conflicted file in your editor
  2. Look for the conflict markers (<<<<<<<, =======, >>>>>>>)
  3. Decide which version to keep (or combine them)
  4. Delete the conflict markers
  5. Save, stage, and commit
git add conflicted-file.py
git commit -m "Resolved merge conflict in IP whitelist"

Undoing Mistakes

Unstage Files

Accidentally staged the wrong file?

git reset filename.py
# or
git reset # unstage all

Undo Last Commit

Keep changes, just undo the commit:

git reset HEAD~1

Discard Local Changes

Throw away all uncommitted changes:

git checkout -- filename.py
# or
git restore filename.py

Go Back in Time

Completely reset to a previous commit:

git log # find commit hash
git reset --hard abc123

‚ö†Ô∏è This deletes changes!

The .gitignore File

Some files should NEVER be committed to Git (especially in security work!):

‚ö†Ô∏è Never Commit These:

  • API keys and passwords
  • SSH private keys
  • Database credentials
  • Client sensitive data
  • Exploit code that could be weaponized

Create a .gitignore file:

# Create the file
touch .gitignore

# Example .gitignore content:
*.key # All .key files
credentials.txt # Specific file
.env # Environment variables
secrets/ # Entire directory
*.log # Log files
.DS_Store # Mac system files

Git will ignore these files automatically!

Best Practices for Security Professionals

  1. Commit often, push regularly - Don't wait weeks to commit
  2. Write descriptive commit messages - Future you will thank you
  3. Use .gitignore for sensitive data - Never commit credentials
  4. Review code before merging - Use pull requests for team review
  5. Branch for new features - Keep main branch stable
  6. Pull before you push - Avoid conflicts

‚ö†Ô∏è Security Warning:

If you accidentally commit sensitive data (API keys, passwords), simply removing it in a new commit is NOT enough! The data remains in Git history. You need to:

  • Use tools like git filter-branch or BFG Repo-Cleaner
  • Rotate the compromised credentials immediately
  • Consider the data already exposed

Common Mistakes to Avoid

‚ùå Committing to Wrong Branch

Always check which branch you're on:

git branch # Shows current branch

‚ùå Not Pulling Before Push

Can cause conflicts and rejected pushes. Always:

git pull
# then push

‚ùå Vague Commit Messages

Bad: "fixed stuff"
Good: "Fixed XSS vulnerability in search form"

‚ùå Committing Secrets

Use .gitignore and environment variables!

# Use .env files
API_KEY=secret123

Viewing History & Logs

Git keeps a complete history. Use it for auditing and investigation:

# View commit history
git log

# Compact view (one line per commit)
git log --oneline

# See who changed what in a file
git blame filename.py

# View changes in a specific commit
git show abc123

# Search commit messages
git log --grep="vulnerability"

Security Use Case:

Investigating when a vulnerability was introduced:

git log --all --grep="authentication" -- login.py

This shows all commits related to authentication in the login.py file.

Practice Exercise

Create Your Security Portfolio Repository

  1. Create a new repository on GitHub called "security-portfolio"
  2. Clone it to your local machine
  3. Create a README.md describing your cybersecurity interests
  4. Add a .gitignore file
  5. Create a "scripts" folder with a simple Python script (e.g., port scanner)
  6. Commit and push everything to GitHub
  7. Create a new branch called "feature/add-network-tool"
  8. Add another script in this branch
  9. Push the branch and create a Pull Request on GitHub
  10. Merge the PR

This will be your foundation for showcasing security projects to potential employers!
